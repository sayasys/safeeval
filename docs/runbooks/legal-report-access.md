# Runbook -- Legal-audience report access (manual ops MVP)

**Status:** active (report-gen Phase 3).
**Owner:** Trust & Safety lead / ops on-call.
**Gate implementation:** `src/lib/report-generators/legal-auth-gate.ts` (`requireLegalAccess`).
**Audit table:** `legal_access_log` (migration `src/lib/data/schema/M10_legal_access_log.sql`).
**Route:** `GET /api/app/reports/:evaluation_id/legal`.

This runbook documents the manual, MVP access process for the legal-audience
report. It is intentionally manual: at MVP there is no self-service grant flow.
Phase 4+ replaces the manual role assignment with real RBAC (see the bottom of
this doc); the audit-log behavior persists across that upgrade.

## When and why legal-audience reports are generated

The report generator translates a persisted, PII-sanitized evaluation envelope
into an audience-specific markdown report. The `legal` audience is the
register tuned for legal / compliance counsel reviewing a case for regulatory
exposure -- predominantly `block` and high-severity `human_review`
dispositions. The legal report carries:

- The regulatory framework mapping (IC3 / FTC / NIST / FinCEN category in the
  framework's own vocabulary).
- Chain-of-custody audit metadata (stage prompt hashes, `cache_key`,
  `ontology_version`, `schema_version`, evaluation timestamp).
- The disposition-rule lockstep reference.
- The access-control record (this very audit trail).

Unlike the `reviewer`, `trust_safety_lead`, and `exec_summary` audiences, the
legal report is **not** pre-generated by the background post-write hook. It is
generated on demand, at read time, only after the access gate passes -- the
background hook has no authenticated user context and would deny every call.

## Who can request access (closed-set roles)

Access is gated on a single coarse role at MVP:

| Role | Source | Grants |
|---|---|---|
| `pii_reviewer` | Supabase Auth `app_metadata.role` (admin-only) | Read legal-audience reports |
| `admin` | Supabase Auth `app_metadata.role` (admin-only) | Force-regenerate any report (POST) |
| (none / any other) | -- | Denied; the attempt is logged |

Eligible humans for the `pii_reviewer` role: counsel, compliance leads, and
authorized fraud-investigation personnel with a documented business need
(regulatory exposure analysis, audit response, chain-of-custody collection).
The role is **not** self-service and **not** derivable from any other
attribute; ops assigns it explicitly per the process below.

Why `app_metadata` and not `user_metadata`: in Supabase Auth, `app_metadata`
is writable only with the service-role key (the admin panel / admin API),
whereas `user_metadata` is writable by the end user themselves. Storing the
access role in `user_metadata` would let any signed-in user self-escalate to
`pii_reviewer`. The gate reads `app_metadata.role` first for exactly this
reason; `user_metadata.role` is consulted only as a local-development
convenience and must never be relied on in production.

## Manual ops process

### Granting the role

1. **Requester files a ticket.** Subject: "Legal-audience report access --
   evaluation_id=<ID>". Body identifies the case, the regulatory framework the
   access supports (IC3 / FTC / NIST / FinCEN), the business need, and the
   expected read window.
2. **Reviewer (T&S lead or above) approves or rejects** within one business
   day. Approval is recorded in the ticket; rejection includes a rationale.
3. **On approval, ops sets the role** in the Supabase Auth admin panel:
   open the user, edit **App Metadata** (not User Metadata), and set
   `{ "role": "pii_reviewer" }`. The change takes effect on the user's next
   session token refresh (sign out / sign in if immediate access is needed).
4. **Requester accesses the report** via the UI or directly:
   `GET /api/app/reports/<evaluation_id>/legal`. The gate verifies the role,
   writes a `granted=true` row to `legal_access_log`, and returns the markdown.
5. **Audit captured automatically.** Every access attempt -- granted or denied
   -- writes a `legal_access_log` row (`user_id`, `evaluation_id`,
   `audience='legal'`, `granted`, `denied_reason`, `accessed_at`). No manual
   log step is required.

### Revoking the role

When the documented business need ends, ops clears the role: in the admin
panel, edit the user's **App Metadata** and remove the `role` key (or set it to
a non-privileged value). The next token refresh drops access. Revocation does
not delete prior `legal_access_log` rows -- the audit trail is immutable by
design.

### Escalation (same-day)

If the requester believes the case meets a regulatory-disclosure deadline that
cannot wait one business day, escalate via the on-call T&S lead pager.
Same-day approval is supported via this path; the role-grant and audit steps
are otherwise identical.

## Rotation and review cadence

- **Quarterly grant review.** Once per quarter, ops exports the current set of
  users carrying `app_metadata.role = 'pii_reviewer'` and confirms each still
  has an active, documented business need. Stale grants are revoked.
- **Monthly denied-access review.** Once per month, security review queries
  `SELECT * FROM legal_access_log WHERE granted = false ORDER BY accessed_at
  DESC` to surface attempted access by users without the role. A cluster of
  denials against one evaluation_id, or repeated denials from one user_id, is
  investigated.
- **Granted-access spot check.** A high `granted` count for a single
  evaluation_id (many reads of the same legal report) is a workflow-efficiency
  or process signal worth a look; flag evaluations with an unusual read volume.

## Future RBAC upgrade (Phase 4+)

This runbook documents the manual MVP. The Phase 4+ RBAC implementation
replaces the manual `app_metadata.role` assignment (grant/revoke steps above)
with an automated role / grant system -- e.g. a `legal_access_grants` table of
time-bounded, single-evaluation-scoped grants issued through an approval flow.
The `requireLegalAccess(user, evaluation_id)` function signature is the stable
boundary: its body swaps from a role check to a grant lookup without touching
the route or the dispatcher. The `legal_access_log` audit write persists
unchanged across the upgrade -- it is the durable compliance record regardless
of how the grant decision is made.

## Example case studies (ops reference)

### Case 1 -- Approved access, regulatory exposure analysis

Counsel files a ticket referencing `evaluation_id=8842`, a `block` disposition
on a wire-transfer-fraud pattern, citing an IC3 referral in progress. The T&S
lead approves; ops sets `app_metadata.role = "pii_reviewer"` on counsel's
account. Counsel requests `GET /api/app/reports/8842/legal` and receives the
markdown; `legal_access_log` records `granted=true`. After the IC3 filing
closes, ops revokes the role at the quarterly review.

### Case 2 -- Denied access, role not provisioned

A fraud analyst (not provisioned for `pii_reviewer`) requests
`GET /api/app/reports/8842/legal` directly, expecting the reviewer report.
The gate denies: HTTP 403 with the runbook reference, and a `legal_access_log`
row with `granted=false`, `denied_reason="caller role 'none' lacks
pii_reviewer"`. The monthly denied-access review surfaces the row; security
confirms it was an honest mistake (the analyst wanted the `reviewer` audience,
which needs no gate) and no action is taken beyond noting it.

### Case 3 -- Same-day escalation under a disclosure deadline

Compliance receives a regulator request with a 48-hour response window on a
case landing late Friday. Rather than wait for the one-business-day approval,
compliance pages the on-call T&S lead, who approves same-day. Ops sets the role
out-of-hours; compliance pulls `GET /api/app/reports/<ID>/legal`. The grant is
back-documented in the ticket on the next business day, and the
`legal_access_log` row already captured the access at request time.
